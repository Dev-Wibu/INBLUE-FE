## §1 — Hard Rules

These rules are **non-negotiable**. Violating any of them is a blocking error.

- **Vietnamese UI**: All user-facing strings (labels, buttons, toasts, placeholders, validation messages) **must** be in Vietnamese.
- **Never edit `schema-from-be.d.ts`**: This file is auto-generated from the backend OpenAPI spec. Regenerate it with `pnpm generate-schema`.
- **Quality gate**: Run `pnpm validate` before finishing any task. This chains `format:check → lint → typecheck → build`. All steps **must** pass.
- **Path alias**: `@/` maps to `src/` (configured in `vite.config.ts` and `tsconfig.json`). Always use `@/` imports, never relative `../../`.
- **No AI attribution**: NEVER add `Co-authored-by`, `Generated by`, or any AI attribution to git commits. All commits must belong to the human user.
- **No `pnpm-lock.yaml` edits**: Never manually modify `pnpm-lock.yaml`. It is updated automatically by `pnpm install`.

### Comment Rules

> Good code is self-explanatory. Only write comments when the code **cannot speak for itself**.

Write a comment only for one of these three reasons:

1. **Complex logic** — explain an algorithm or non-obvious processing step
2. **Risk warning** — the action can cause errors or unintended side-effects
3. **Explain why** — why the code is written this way, not what it does

```ts
// ✅ Complex logic
// Tiered discount: <5 tickets → 0%, 5–10 → 10%, >10 → 20%
const finalPrice = calculateTieredDiscount(quantity, unitPrice);

// ✅ Risk warning
// WARNING: Clearing this cache will log out all active users
clearAuthCache();

// ✅ Explain why
// setTimeout(0) ensures DOM has rendered before focusing
setTimeout(() => inputRef.current?.focus(), 0);

// ❌ Redundant — code is already clear
const isLoggedIn = !!token; // check login ← unnecessary
```

## §3 — API Layer

### `$api` — Canonical (use for all new code)

Type-safe client built on `openapi-fetch` + `openapi-react-query`, generated against `schema-from-be.d.ts`.

```typescript
import { $api } from "@/lib/api";

// Queries
$api.useQuery("get", "/api/sessions");
$api.useQuery("get", "/api/sessions/{id}", { params: { path: { id } } });

// Mutations
$api.useMutation("post", "/api/sessions");
$api.useMutation("put", "/api/sessions/{id}");
$api.useMutation("delete", "/api/sessions/{id}");
```

**How it works** (`src/lib/api.ts`):

- Base URL: `VITE_API_BASE_URL` (defaults to `https://api.kdz.asia`)
- JWT auto-injected via `onRequest` middleware (reads from `useAuthStore`)
- `onResponse` middleware unwraps backend's `{ traceId, data }` wrapper → returns raw `data`
- 401 responses auto-logout: clears `auth-storage`, redirects to `/login`
- Error normalization via `normalizeApiError()` with Vietnamese messages
- Skips auth header for public endpoints (`isPublicAuthRequest()`)

### Axios Managers — Legacy (do NOT create new ones)

Class-based services in `src/services/*.manager.ts`. Each implements `BaseManager<T>` and uses `createApiInstance()` from `constants/api.config.ts`.

```typescript
// ⚠️ LEGACY — do not create new managers
import { sessionManager } from "@/services";
const response = await sessionManager.getAll();
```

**How `createApiInstance()` works** (`src/constants/api.config.ts`):

- Base URL: `VITE_API_BASE_URL` (fallback `http://localhost:8080`)
- Request interceptor: auto-attaches Bearer token from `useAuthStore`, adds `X-Request-ID`
- Response interceptor: unwraps `traceId` wrapper, handles 401 → clearAuth + redirect
- Timeout: 90 seconds

> **Note**: Both `$api` and Axios managers auto-inject JWT and handle 401 logout. The old instructions incorrectly stated Axios doesn't inject JWT — this has been corrected.

### Migration Checklist (Axios → $api)

When you encounter Axios manager code during a feature change:

1. Check if the endpoint exists in `schema-from-be.d.ts`
2. If yes, replace the manager call with `$api.useQuery()` or `$api.useMutation()`
3. Update the hook to use `$api` directly instead of wrapping the manager
4. Remove unused manager methods (if the manager becomes empty, delete it)
5. Remove barrel export from `services/index.ts`
6. Update tests accordingly

### Error Handling

- `constants/api.config.ts` exports `ERROR_MESSAGES` — a `Record<number, string>` mapping HTTP status codes to Vietnamese messages
- `lib/error-normalizer.ts` exports `normalizeApiError()` for consistent error objects with `traceId`, `message`, `status`
- `lib/toast-service.ts` (if used) or `sonner` directly for toast notifications
- `buildEndpoint()` from `constants/api.config.ts` substitutes `:param` placeholders: `buildEndpoint("/api/users/:id", { id: 5 })`
- `API_ENDPOINTS` — comprehensive const object with all endpoint paths grouped by domain

## §4 — State Management

### Server State → React Query

All server-fetched data is managed through `@tanstack/react-query`.

```typescript
// Import the shared query client
import { queryClient } from "@/lib/queryClient";

// Invalidate after mutation
queryClient.invalidateQueries({ queryKey: ["get", "/api/sessions"] });
// openapi-react-query keys use ["method", "/path"] format
```

**Configuration**: `QueryProvider` in `src/contexts/QueryProvider.tsx` wraps the entire app.

### Client State → Zustand

Zustand with `persist` middleware + `localStorage`. The project does **not** use React Context for state (only `QueryProvider` context exists).

**Stores** (`src/stores/`):
| Store | Purpose |
|---|---|
| `authStore` | JWT token, user info, login state, session expiry. Storage key: `auth-storage` |
| `notificationStore` | Notification preferences and unread counts |
| `settingsStore` | User preferences and app settings |
| `themeStore` | Dark/light theme state |

**Canonical example**: `stores/authStore.ts` — demonstrates `create()` + `persist()` + `partialize()` + `onRehydrateStorage()`.

**Pattern**:

```typescript
import { create } from "zustand";
import { createJSONStorage, persist } from "zustand/middleware";

export const useMyStore = create<MyState>()(
  persist(
    (set) => ({
      // state + actions
    }),
    {
      name: "my-storage",
      storage: createJSONStorage(() => localStorage),
      partialize: (state) => ({
        /* only persist these fields */
      }),
    }
  )
);
```

## §8 — Adding a New Feature

1. `pnpm generate-schema` — regenerate types if backend changed
2. Add/update types in `interfaces/schema.types.ts` (or use generated types from `schema-from-be.d.ts` directly)
3. **New code**: Use `$api.useQuery()` / `$api.useMutation()` directly in a custom hook. **Do NOT create new Axios managers.**
4. Create hook in `hooks/` wrapping the API calls with business logic
5. Create page in `pages/<Role>/` following existing folder structure
6. Add route in `App.tsx`:
   - **User/Mentor**: nested under their ChromeTabs shell
   - **Admin**: add tab to `AdminDashboardPage` (no new route)
   - **Staff**: add standalone route at `/staff/<feature>`
7. Export from barrel files (`hooks/index.ts`, etc.)
8. Run `pnpm validate` — fix all errors before finishing

## §10 — Gotchas

- **`schema-from-be.d.ts` is the contract**: All type-safe API calls depend on this file. After backend deploys a breaking change, run `pnpm generate-schema` before coding.
- **Admin pages are NOT routed** — they render via tab switching inside `AdminDashboardPage`. Adding an admin feature means adding a tab + component, not a route.
- **`buildEndpoint()` syntax**: `buildEndpoint("/api/users/:id", { id: 5 })` substitutes `:param` placeholders.
- **Form validation**: Zod schemas + react-hook-form (`@hookform/resolvers`). Never use `formik`.
- **`JoinSessionRequest.isMentor`**: Must be explicitly `true` or `false` — backend cannot deserialize `null`/`undefined` to boolean.
- **Prettier auto-sorts imports**: Don't add manual import grouping comments — they'll be removed on `pnpm format`.
- **Backward-compat redirects**: Many old routes (`/dashboard/*`, `/mentor-dashboard/*`, `/admin/dashboard`, etc.) redirect to new paths. Check `App.tsx` before adding similar routes.
- **Backend `traceId` wrapping**: API responses come wrapped as `{ traceId, data }`. Both `$api` and `createApiInstance()` auto-unwrap this — you receive raw `data`.
- **Session expiry**: `SessionExpiryGuard` component polls every 30s + on visibility/focus change. It reads `expiresAt` from `authStore` and auto-logouts.
- **Two base URL defaults**: `$api` defaults to `https://api.kdz.asia`; `createApiInstance()` defaults to `http://localhost:8080`. Both read from `VITE_API_BASE_URL`. Ensure `.env` is set correctly.

## §11 — Anti-Patterns & Code Quality

These anti-patterns were catalogued in `docs/error.md`. **Do NOT introduce them in new code. Fix them when encountered in existing code.**

### ❌ AP-01: Dead Code & Unused Files

- Do NOT leave files that are not imported anywhere in `src/`. Delete them.
- Do NOT leave commented-out code blocks (`//`, `/* */`). Delete them — use git history.
- Do NOT keep packages in `package.json` that have zero imports (e.g., `formik`, `react-icons`).
- Do NOT keep mock data files that are no longer imported.

### ❌ AP-02: DRY Violations (Code Duplication)

- Do NOT define `formatDate` / `formatDateTime` / `formatCurrency` inline. **Always import from `@/lib/formatting`**.
- Do NOT define inline `getStatusBadge*` helpers. Use `<StatusBadge>` from `@/components/shared/StatusBadge`.
- Do NOT copy-paste ChromeTabs or Sidebar implementations per-role. Use `DashboardChromeTabs` / `DashboardSidebar` from `@/components/shared/`.
- If a UI pattern appears in ≥2 files, extract it into a shared component.
- If a logic pattern appears in ≥2 hooks, extract it into a utility function in `lib/`.

### ❌ AP-03: God Components (>500 LOC)

- A component file should stay under **300 LOC**. Files approaching 500 LOC **must** be split.
- Do NOT put step wizard logic, form state, and API calls all in one file. Extract each step into its own component + a dedicated hook for state/API.
- Do NOT mix unrelated domain concerns in one component.
- If a component has >7 `useState` calls or multiple complex `useEffect` blocks, it must be refactored.

### ❌ AP-04: Misleading Names

- File name must match its primary export and actual functionality.
- Do NOT name a file `PaymentPage` if it contains no payment logic.
- Do NOT put utility functions inside `mocks/` files. `mocks/` is for test/mock data only.
- Export name must match file name: `UserCard.tsx` must export `UserCard`, not `ProfileWidget`.

### ❌ AP-05: Misplaced Files

- UI components belong in `components/`, not `utils/` or `hooks/`.
- API logic belongs in `services/` or `hooks/`, not `components/`.
- Shared cross-role components belong in `components/shared/`, not inside a role-specific folder.
- Test files must be colocated with their source file.

### ❌ AP-06: Naming Convention Violations

- See §7. Fix violations (e.g., kebab-case hooks, camelCase utils) when encountered.

### ❌ AP-07: Performance & Re-render Issues

- Do NOT use array index as `key` in `.map()` for dynamic lists (add/remove/sort).
- Do NOT declare heavy functions/objects inside component body without `useMemo` / `useCallback` when passing to child components.
- Do NOT use `setInterval`, `addEventListener`, or subscriptions in `useEffect` without a cleanup return.
- ✅ Always return a cleanup function: `return () => { clearInterval(id); }`.

### ❌ AP-08: Unsafe Async Handling

- Do NOT call `async/await` API functions without `try/catch`.
- Do NOT access deep nested properties without optional chaining: use `data?.user?.profile?.avatar`.
- Do NOT forget loading states — always manage `isLoading` to prevent double-submit.

### ❌ AP-09: Race Conditions in Data Fetching

- Do NOT call `fetch`/`axios` in `useEffect` without `AbortController` or an `ignore` flag.
- ✅ Use React Query's built-in request deduplication and cancellation instead of raw `useEffect` fetching.
- If you must use `useEffect` for fetching, include cleanup:
  ```typescript
  useEffect(() => {
    const controller = new AbortController();
    fetchData({ signal: controller.signal });
    return () => controller.abort();
  }, [dependency]);
  ```

### ❌ AP-10: Derived State Abuse

- Do NOT create `useState` + `useEffect` pairs just to sync one state from another.
- ✅ Calculate derived values directly:

  ```typescript
  // ❌ Bad
  const [filtered, setFiltered] = useState([]);
  useEffect(() => {
    setFiltered(items.filter(predicate));
  }, [items]);

  // ✅ Good — derived state
  const filtered = items.filter(predicate);
  // Or with useMemo if the computation is expensive:
  const filtered = useMemo(() => items.filter(predicate), [items]);
  ```

   ```bash
   curl -s -X POST "https://api.kdz.asia/auth/login" \
     -H "Content-Type: application/json" \
     -d '{"email":"binhan@gmail.com","password":"123"}' | jq .
   ```

3. **Use the token** in subsequent requests:

   ```bash
   curl -s "https://api.kdz.asia/api/sessions" \
     -H "Authorization: Bearer <TOKEN>" | jq .
   ```

4. Always extract the `traceId` from error responses for debugging.

### 12.3 — Validation Loop

When finishing a task, run this loop until clean:

1. Run `pnpm validate`
2. If errors are found:
   a. Parse the error output to identify affected files and error types
   b. Fix all errors (formatting → lint → type errors → build errors, in that order)
   c. Go to step 1
3. Only stop when `pnpm validate` exits with code 0 and all checks pass

**Shortcut for common fixes**:

- Formatting issues: `pnpm format` (auto-fix)
- Lint issues: `pnpm lint:fix` (auto-fix what's possible)
- Type errors: manual fix required
- Build errors: manual fix required

## §13 — Auto-Evolution Protocol

This section defines when and how this `rules.md` file must be automatically updated to stay synchronized with the actual codebase.

### Trigger Conditions

The AI agent **MUST** update this file when ANY of the following changes occur:

| Trigger                                                       | What to update                                |
| ------------------------------------------------------------- | --------------------------------------------- |
| **Dependency added/removed** in `package.json`                | §2 Active/Dead libraries table                |
| **Route added/modified/removed** in `App.tsx`                 | §2 Route trees table, §10 Gotchas if relevant |
| **Zustand store added/modified** in `stores/`                 | §4 Stores table                               |
| **New shared component created** in `components/shared/`      | §5 Shared Components table                    |
| **New custom hook created** in `hooks/`                       | §5 Custom Hooks table                         |
| **API pattern changed** (e.g., new middleware, auth flow)     | §3 API Layer                                  |
| **Auth flow changed** (e.g., OAuth, SSO added)                | §3, §4, §10                                   |
| **Environment variable added/changed** in `.env`              | §10 Gotchas                                   |
| **Build/tooling config changed** (Vite, ESLint, Nx, Prettier) | §9 Commands, §2 Stack                         |
| **New design tokens / color changes**                         | §6 Design Tokens table                        |
| **Service manager migrated to `$api`**                        | §3 Migration progress, §5 if hooks changed    |
| **Test infrastructure changed**                               | §9 Commands, relevant Gotchas                 |

### Update Procedure

When a trigger is detected:

1. **Identify** which section(s) of `rules.md` are affected
2. **Update** the affected sections to reflect the new state of the codebase
3. **Update** `@LAST_SYNCED` timestamp at the top of this file
4. **Mirror** the exact same changes to `.github/copilot-instructions.md`
5. **Append** a changelog entry to §14 describing what changed

### Rules for Self-Modification

- **Never remove a section** — only update content within sections
- **Never change the section numbering** (§1–§14) — only add sub-sections if needed
- **Keep it concise** — if a table row is removed, don't leave empty rows
- **Preserve examples** — update code examples to reflect the current API, don't delete them
- **Atomic updates** — update both `rules.md` and `copilot-instructions.md` in the same commit
